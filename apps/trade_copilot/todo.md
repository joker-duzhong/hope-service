### 🤖 第一部分：发给 AI 的开发 Prompt（直接复制发送）

> **【系统提示词】**
> 你是一个资深的 Python 量化开发工程师。我们的项目使用 FastAPI 框架，现在需要你基于 `akshare` 库封装几个核心的数据获取服务类。请务必处理好网络异常重试（使用 `tenacity` 库）、数据格式转换（统一转为标准的 List[dict] 或 Pydantic Model），并尽量利用异步或缓存避免频繁请求。
>
> **【需要实现的接口能力清单】**
> 
> 1. **获取 A 股实时盘口快照（用于计算实时止损/止盈）**
>    *   **依赖接口：** `akshare.stock_zh_a_spot_em()`
>    *   **功能要求：** 获取当前全市场的实时行情。要求能传入一个 `symbol_list` (如 `['600519', '000001']`)，返回这些特定股票的当前价格、涨跌幅、最新时间。
> 
> 2. **获取指数历史日 K 线（用于判断大盘红绿灯）**
>    *   **依赖接口：** `akshare.index_zh_a_hist(symbol="000001", period="daily")` （上证指数）和 `symbol="399001"`（深证成指）
>    *   **功能要求：** 获取最近 30 个交易日的收盘价和成交量，并计算出当天的 **20日均线（MA20）**。返回一个包含“日期、收盘价、MA20、是否跌破20日线”的字典。
> 
> 3. **获取个股历史日 K 线（用于观察池技术形态打分）**
>    *   **依赖接口：** `akshare.stock_zh_a_hist(symbol="股票代码", period="daily", adjust="qfq")` （注意：必须是前复权 `qfq`）
>    *   **功能要求：** 获取指定股票近 60 个交易日的 K 线，计算 MA5, MA10, MA20。
> 
> 4. **获取全市场 ST 股票名单（用于基础排雷）**
>    *   **依赖接口：** `akshare.stock_zh_a_st_em()`
>    *   **功能要求：** 每天只需获取一次，返回一个包含所有 ST 股票代码的 List，供系统比对过滤。

### 📋 第二部分：V1.0 MVP 落地 TODO List (按责任细分)

基于你现有的 FastAPI + Celery 架构，我们将 V1.0 拆分为 3 个小 Sprint。

#### 🎯 Sprint 1: 基础设施与“不亏钱”保镖（移动止盈止损）
**核心目标：把实盘买入的记录录入系统，系统能在盘中通过飞书骂醒你。**

**【Backend Todo】**
- [ ] **Task 1.1: 飞书 Webhook 封装**
  - **描述：** 写一个通用的工具函数，接收文本/Markdown，通过 `httpx` POST 到飞书机器人。
  - **验收标准：** 调用 `send_feishu_alert(title="测试", msg="Hello")`，手机能收到飞书卡片推送。
- [ ] **Task 1.2: 实盘持仓表 (Positions) CRUD**
  - **描述：** 使用 SQLAlchemy 创建 `Position` 模型（字段：代码、名称、买入日期、成本价、买入数量、**盘后最高价(高水位线)**、状态(持仓/已清仓)）。提供增删改查 API。
  - **验收标准：** 可通过 Swagger UI / Postman 成功添加一条持仓记录。

**【Celery Task Todo】**
- [ ] **Task 1.3: 盘中价格监控引擎 (Celery Beat)**
  - **描述：** 注册一个每 5 分钟执行一次的定时任务（限交易日 9:30-11:30, 13:00-15:00）。读取数据库中状态为“持仓”的股票，调用 AkShare 实时接口比对价格。
  - **逻辑：** 
    1. 如果 `当前价 < 成本价 * 0.95` (绝对止损-5%) -> 触发飞书【红色强制割肉】警告。
    2. 如果 `当前价 > 历史最高价` -> 更新 DB 中的历史最高价。
    3. 如果 `当前价 < 历史最高价 * 0.92` (高位回撤8%) -> 触发飞书【橙色止盈落袋】警告。
  - **验收标准：** 手动在数据库造一个破位的数据，启动 Celery Worker，5分钟内收到飞书警告。

**【Frontend Todo】**
- [ ] **Task 1.4: 持仓管理界面**
  - **描述：** 一个简单的表格页，展示当前持仓，提供“新增买入”、“标记卖出”按钮。
  - **验收标准：** 前端能与后端 FastAPI 成功联调，展示持仓列表。

### 🚀 V2.0 / 进阶演进与优化建议 (Ideas Log)

> 在 V1.0 MVP 的后端开发回归完毕后，整个基础大底座已经非常扎实了。以下是对未来阶段（如 V1.1 或 V2.0）的演进思考，供量化系统下一波升级参考：

#### 🔧 1. 风控与交易规则动态化 (Configuration via DB)
*   **痛点**：现在的 `-5%` 绝对止损、`-8%` 回撤止盈、是否双指数破 20日线等参数，都是硬编码在代码或 Celery Task 中的。
*   **优化**：建立 `TradeStrategySettings` 动态规则表，将【止损滑点比例】、【止盈回撤比例】、【MA 均线参(比如改为 MA30)】等参数全面表驱动化，或者在 `Position` 单独针对不同股性设定各自的止损位，更灵活地去应对高波动票（如柚子票与白马票分开放宽）。

#### 💸 2. 仓位管理与凯利公式算盘 (Position Sizing)
*   **痛点**：当前系统只告诉你“什么时候该跑、今天能不能买”，并没有介入核心的“买多少”。
*   **优化**：在狙击雷达或买入推荐时，增加一层【凯利算盘引擎】或【ATR 波动率算盘】。依据大盘目前的红绿灯状态（如：如果是橘灯，则建议最大仓位控制在 20%），在发出的卡片上直接计算出**“建议分配资金：XXXX元，建议买入：XXX手”**。

#### 📊 3. 板块轮动与市场温度计
*   **演进**：现有的 `MarketService` 只有两指大盘。可以扩展抓取【申万一级行业】或【全市场涨跌停家数/连板高度】，建立包含情绪周期的“赚钱效应打分”。从而在“大盘走弱”的橙灯期也能精准捕捉到热点主线的突破！

#### 📝 4. 交易日记 (Trading Journal)
*   **痛点**：*“我们写代码，不是为了预测未来，而是为了在意外发生时，机器能逼着我们执行纪律。”* 量化的核心是克服人性弱点，如果只依赖冰冷的买卖信号而忽略了人工操作反思，容易重犯主观错误。
*   **优化**：提供一个用户手动填写的【每日复盘日志】模块。记录每天有没有按系统指示闭眼操作（执行力打分）、有没有主观犯错、情绪控制的如何。把系统的“纪律”与人的“反思”结合起来，形成交易闭环。
